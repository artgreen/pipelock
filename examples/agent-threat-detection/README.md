# Agent Threat Detection: OTLP + Signed Receipt Example

This directory shows the join between Pipelock's OTLP runtime telemetry and its signed action receipts when the unstable OTel [`agent.threat.detection.*`](https://github.com/open-telemetry/semantic-conventions-genai/issues/132) semantic convention is enabled.

**Three artifacts, one operation, joined by `correlation_id`:**

| File | What it is | Where it would live in production |
|---|---|---|
| `otlp-log-record.json` | The OTLP log record Pipelock exports to a collector when `emit.otlp.agent_threat_detection_emit: true`. Carries the convention attributes alongside Pipelock-native fields. | Your OTLP collector or SIEM. |
| `signed-receipt.json` | The signed action receipt Pipelock writes to its audit-packet store for the same operation. Cryptographically verifiable offline. | Long-term audit store + receipt verifier. |
| `README.md` | This walkthrough. | n/a |

## The join

Both files share the same `correlation_id` (in the OTLP record) / `request_id` (in the receipt):

```text
req-2026-05-13-9c4ad1
```

A consumer that wants the full picture for an operation does the join in two steps:

1. Find the OTLP log record with the matching `agent.threat.detection.correlation_id`.
2. Load the receipt with that `request_id` from the audit store and verify its signature against the Pipelock signer's public key. Verifiers exist today in Go (`cmd/pipelock-verifier`), TypeScript (`sdk/verifiers/ts`), and Rust (`sdk/verifiers/rust`).

The OTLP record gives you the runtime view; the receipt gives you the cryptographic proof.

## Convention attributes in the OTLP record

The five proposed attributes are:

| Attribute | Value here | Meaning |
|---|---|---|
| `agent.threat.detection.rule_id` | `example-bundle-rule-001` | The matched rule identifier, scoped by `ruleset`. |
| `agent.threat.detection.ruleset` | `pipelock-rules@1.2.3` | Source-specific identity. `pipelock-rules@<bundle-version>` for bundle-sourced detections; `pipelock-core@<binary-version>` for built-in scanner detections. |
| `agent.threat.detection.severity` | `high` | Mapped from Pipelock's internal 3-level severity. |
| `agent.threat.detection.action` | `block` | The enforcement decision at detection time. |
| `agent.threat.detection.correlation_id` | `req-2026-05-13-9c4ad1` | Opaque, producer-defined. Matches `request_id` in the signed receipt. |

See [`docs/observability/agent-threat-detection.md`](../../docs/observability/agent-threat-detection.md) for the full mapping table, severity vocabulary, and configuration state matrix.

## Caveats

* The proposed convention is unstable. Attribute names may change in subsequent Pipelock releases until the OTel SIG accepts the proposal.
* The signed receipt shape in this example follows Pipelock's Audit Packet v0 schema (`sdk/audit-packet/v0/`). For cryptographically valid signature samples that round-trip through the published verifiers, see `sdk/conformance/testdata/`.
* `correlation_id` is the opaque `request_id`, not a content-addressed hash of the receipt. Hash form is intentionally not exposed in v0 to avoid coupling the convention to receipt internals and to keep emission cheap (the receipt is signed after the log record is emitted).

## How this example was generated

The OTLP log record is produced by the deterministic golden sample test in `internal/emit/otlp_agent_threat_test.go`. The signed receipt is a synthetic but schema-valid example. Both can be regenerated by running:

```sh
go test -run AgentThreatDetectionAttrs_GoldenSample ./internal/emit/...
```
